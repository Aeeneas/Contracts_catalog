## Техническое задание на разработку портала-каталогизатора документов (ФИНАЛЬНАЯ ВЕРСИЯ)

**1. Наименование проекта:** Портал-каталогизатор договорной документации.

**2. Цель проекта:**
Создание автоматизированного портала для стандартизации хранения, каталогизации и интеллектуального анализа договорной документации (договоры, закрывающие документы, акты) с использованием веб-интерфейса, базы данных и функций искусственного интеллекта.

**3. Задачи проекта:**
*   Разработка стандартизированной структуры файлового хранилища для договорной документации.
*   Автоматическая организация существующих и вновь загружаемых документов по заданной структуре.
*   Создание базы данных для хранения метаданных договоров.
*   Разработка веб-интерфейса для загрузки, поиска и просмотра договоров.
*   Интеграция с функциями искусственного интеллекта (ИИ) для автоматического извлечения данных из документов и генерации кратких описаний.
*   Реализация механизма проверки дубликатов договоров.
*   Реализация генерации и присвоения уникальных сквозных номеров договорам.

**4. Функциональные требования:**

**4.1. Автоматическая организация файлового хранилища:**
*   **Стандартизация структуры:** Документы должны быть организованы по следующей иерархии:
    `[Компания]\[Заказчик]\[Тип работ]\[Год]`
    *   `Компания`: Юридическое лицо - **исполнитель** договоров (одно из: "ТОР-ЛИФТ", "Противовес", "Противовес-Т").
    *   `Заказчик`: Наименование заказчика.
    *   `Тип работ`: Возможные значения: "ТО" (Техническое обслуживание), "МОНТАЖ", "СТРОЙКА", "ПРОЕКТИРОВАНИЕ", "КАПИТАЛЬНЫЕ РАБОТЫ".
    *   `Год`: Год заключения или действия договора.
*   **Перемещение документов:** Автоматическое перемещение загруженных документов (договоры, приложения, акты, счета-фактуры, КС-2, КС-3 и т.п.) в соответствующую конечную папку согласно новой структуре.
*   **Обработка ZIP-архивов:** Система должна автоматически распаковывать `.zip` архивы и обрабатывать их содержимое как отдельные документы. После успешной обработки оригинальный `.zip` файл **должен быть перемещен в специальную архивную папку**.
*   **Обработка неинформативных папок:** В случае обнаружения папок с неинформативными названиями (например, "Новая папка"), система должна предпринять попытку извлечения информации (Компания, Заказчик, Тип работ, Год) из документов, находящихся внутри этих папок, используя приоритетный ИИ-анализ.
*   **Обработка года с суффиксом "г":** Система должна корректно парсить год из названия папки, даже если он содержит суффикс "г" (например, `2020г` -> `2020`).

**4.2. База данных договоров (метаданные):**
*   **Схема данных:** Создание реляционной базы данных с таблицей для хранения следующей информации по каждому договору:
    *   `ID` (Primary Key, уникальный идентификатор договора)
    *   `Уникальный_номер_договора` (Текст, генерируется системой по заданному формату)
    *   `Компания` (Текст, юридическое лицо - **исполнитель** договоров: "ТОР-ЛИФТ", "Противовес", "Противовес-Т")
    *   `Заказчик` (Текст)
    *   `Тип_договора/работ` (Текст, "ТО", "МОНТАЖ", "СТРОЙКА", "ПРОЕКТИРОВАНИЕ", "КАПИТАЛЬНЫЕ РАБОТЫ")
    *   `Стоимость_договора` (Числовое поле, например, десятичное с двумя знаками после запятой)
    *   `Стоимость_в_месяц` (Числовое поле, опционально)
    *   `Этапы_выполнения` (Текст/JSON. **Если ИИ не выделяет явные этапы, это поле должно содержать значение, указывающее на одноэтапное выполнение (например, текстовая строка "Один этап").**)
    *   `Краткое_описание_договора` (Текст, генерируется ИИ)
    *   `Дата_заключения` (Дата)
    *   `Дата_начала_действия` (Дата. **Если не найдено явно в документе, устанавливается равной `Дата_заключения`.**)
    *   `Дата_окончания_действия` (Дата. **Если не найдено явно в документе, устанавливается как `Дата_начала_действия` + 12 месяцев.**)
    *   `Путь_к_каталогу` (Текст, путь к папке с документами договора в файловой системе)
    *   `Дата_загрузки` (Дата/Время)
    *   `Статус_анализа_ИИ` (Текст, например, "В ожидании", "Завершено", "Ошибка")

**4.3. Веб-интерфейс:**
*   **Главная страница (Поиск и Просмотр):**
    *   Поле поиска по всем текстовым полям БД, включая `Уникальный_номер_договора`.
    *   Фильтры по полям: `Компания`, `Заказчик`, `Тип договора/работ`, `Год`, `Дата заключения` (диапазон), `Уникальный_номер_договора`.
    *   Табличное отображение списка договоров с возможностью сортировки по колонкам.
    *   Для каждой записи в таблице:
        *   Краткая информация (Компания, Заказчик, Тип, Описание, Стоимость, Даты).
        *   Кнопка/ссылка для открытия соответствующей папки с документами в файловой системе.
        *   Кнопка/ссылка для просмотра детальной информации по договору.
*   **Страница "Добавить договор":**
    *   Область Drag&Drop для загрузки одного или нескольких файлов (PDF, DOCX и т.п.), относящихся к одному договору.
    *   Кнопка "Загрузить и обработать".
    *   Отображение статуса проверки на дубликаты.

**4.4. Интеграция с искусственным интеллектом (ИИ):**
*   **Приоритет ИИ-данных:** Данные, извлеченные ИИ из содержимого документа, имеют наивысший приоритет при определении значений полей `Компания`, `Заказчик`, `Тип_договора/работ`, `Стоимость_договора`, `Стоимость_в_месяц`, `Дата_заключения`, `Дата_начала_действия`, `Дата_окончания_действия` и `Этапы_выполнения`. Если данные, извлеченные ИИ, противоречат данным, полученным из имени файла/пути, используются данные ИИ.
*   **Извлечение данных:** Использование API нейросети (например, Gemini API, после его OCR-обработки для сканов) для автоматического извлечения следующих данных из текста загруженных документов:
    *   `Компания`
    *   `Заказчик`
    *   `Тип договора/работ` (ИИ должен различать виды работ, такие как "МОНТАЖ" и "КАПИТАЛЬНЫЕ РАБОТЫ", основываясь на описании характера работ в документе).
    *   `Стоимость_договора` и `Стоимость_в_месяц`
    *   `Дата_заключения`, `Дата_начала_действия`, `Дата_окончания_действия`.
    *   Информация об `Этапах_выполнения`.
*   **Суммаризация договора:** Автоматическая генерация краткого, информативного описания сути договора для поля `Краткое_описание_договора`.

**4.5. Автоматизированный процесс обработки документов:**
После загрузки документов через веб-интерфейс (Drag&Drop) должна запускаться следующая последовательность действий:
1.  **Прием файлов:** Загрузка файлов на сервер во временную директорию. Включая автоматическую распаковку ZIP-архивов и последующее перемещение оригинальных ZIP-файлов в специальную архивную папку.
2.  **Анализ файлов (предварительный):**
    *   Определение типа документа (договор, акт, счет и т.д.).
    *   Попытка извлечения предварительных метаданных (Заказчик, Год) из имен файлов и/или текущей структуры папок.
    *   Если документ находится в "неинформативной" папке, обработка документов внутри этой папки будет аналогична обычной загрузке, с упором на ИИ-анализ для извлечения всех полей.
    *   **Извлечение текстового слоя:** Перед передачей документа на ИИ-анализ, система должна выполнить извлечение текстового слоя. Для DOCX-файлов это может потребовать использования специализированных библиотек или утилит. Для PDF-файлов, если они являются сканами, потребуется OCR для получения текста.
3.  **ИИ-анализ содержимого:**
    *   Передача текстового содержимого документа в ИИ (Gemini API) для извлечения всех необходимых полей БД.
    *   Генерация `Краткого_описания_договора` с помощью ИИ.
    *   **Приоритет данных:** Данные, извлеченные ИИ, имеют наивысший приоритет.
    *   **Обработка Этапов:** Если ИИ не выделяет явные этапы выполнения работ из документа, поле `Этапы_выполнения` должно быть заполнено значением "Один этап".
    *   **Обработка Дат:** Если `Дата_начала_действия` не найдена явно, она устанавливается равной `Дата_заключения`. Если `Дата_окончания_действия` не найдена явно, она устанавливается как `Дата_начала_действия` + 12 месяцев.
4.  **Проверка на дубликаты:**
    *   Перед сохранением в БД, система должна проверить наличие дубликатов по комбинации ключевых полей (`Компания` + `Заказчик` + `Тип_договора/работ` + `Дата_заключения` + `Стоимость_договора`).
    *   Если обнаружен дубликат, пользователю должно быть предложено подтвердить добавление или отказаться.
5.  **Генерация уникального номера:**
    *   Система должна сгенерировать `Уникальный_номер_договора` по следующему формату:
        `[Аббревиатура_Типа_работ]-[Аббревиатура_Компании]-[Месяц_подписания][Год_подписания]`
        *   **Аббревиатуры Типа работ:**
            *   ТО -> ТХ
            *   МОНТАЖ -> МН
            *   СТРОЙКА -> СТ
            *   ПРОЕКТИРОВАНИЕ -> ПР
            *   КАПИТАЛЬНЫЕ РАБОТЫ -> КР
        *   **Аббревиатуры Компании (Исполнителя):**
            *   Противовес -> ПВ
            *   Противовес-Т -> ПВТ
            *   ТОР-ЛИФТ -> ТЛ
        *   **Месяц подписания:** Две цифры (например, "07" для июля).
        *   **Год подписания:** Последние две цифры года (например, "26" для 2026 года).
        *   **Пример:** Для договора "Монтаж" от "Противовес" с датой подписания в июле 2026 года, уникальный номер будет: `МН-ПВ-0726`.
6.  **Формирование пути:** На основе извлеченных данных (Компания, Заказчик, Тип работ, Год, приоритет которых определен ИИ-анализом) формируется целевой путь в файловом хранилище.
7.  **Создание директории:** Если целевая директория не существует, она создается.
8.  **Перемещение файлов:** Все обработанные файлы перемещаются из временной директории в целевую.
9.  **Запись в БД:** Все извлеченные, сгенерированные и проверенные данные, а также `Путь_к_каталогу` и `Уникальный_номер_договора`, сохраняются в базу данных.
10. **Обратная связь:** Пользователю отображается статус обработки (успешно/ошибка) и ссылка на созданную запись в БД/каталоге.

**5. Нефункциональные требования:**

*   **Безопасность:** Доступ к порталу должен быть авторизован. Все передаваемые данные должны быть защищены.
*   **Производительность:** Быстрая загрузка файлов, оперативный поиск и отображение информации.
*   **Надежность:** Устойчивость к ошибкам при обработке документов, механизмы логирования и отката.
*   **Масштабируемость:** Возможность расширения базы данных и файлового хранилища, а также добавления новых типов документов и ИИ-функций.
*   **Удобство использования (UX):** Интуитивно понятный и современный веб-интерфейс.

**6. Технологический стек:**

*   **Backend (API & логика):** Python (FastAPI).
*   **Frontend (Веб-интерфейс):** React, Vue.js или Angular (по выбору).
*   **База данных:** PostgreSQL.
*   **ИИ-сервис:** Google Gemini API (для извлечения сущностей, суммаризации, возможно OCR).
*   **Хранение файлов:** Файловая система сервера.
